# 长期记忆对话 Agent 项目进展报告

## 一、项目概述

本项目面向长期多会话场景，实现一个具备记忆写入、存储、检索、更新和辅助问答能力的
对话 Agent。系统从历史对话中提取可复用的记忆单元，在后续问题到来时检索相关记忆并
辅助生成答案，从而缓解大语言模型缺乏跨会话持久状态的问题。

本项目主要参考 Mem0 的工程化记忆流水线和 Generative Agents 的检索思路。当前选定的
核心探索方向为**记忆更新机制**：对比只追加写入与冲突感知更新，分析旧信息被新信息
替代时的处理效果。时间允许时，将补充三因子检索作为扩展消融实验。

## 二、当前进度

### 2.1 已完成的系统模块

目前已完成长期记忆 Agent 的主体框架和核心实现：

| 模块 | 完成情况 | 说明 |
|---|---|---|
| Memory Store | 已完成 | 使用向量索引和 JSON 元数据存储记忆；优先使用 FAISS，无法安装时自动回退至 NumPy 索引 |
| Memory Writer | 已完成 | 在会话结束后调用 LLM 批量提取持久化记忆，保存来源 turn id 和重要性 |
| Memory Retriever | 已完成 | 支持基于余弦相似度的 dense top-k 检索 |
| Memory Updater | 已完成 | 支持 `append_only` 和 `conflict_aware` 两种模式 |
| Agent Controller | 已完成 | 完成写入、更新、检索、生成和日志记录的流程编排 |
| Trace Logger | 已完成 | 保存原始对话、派生记忆、检索结果、完整 prompt、回答和响应时间 |
| 数据加载器 | 已完成 | 同时兼容官方 LoCoMo 和助教简化数据格式 |
| 助教评测适配层 | 已完成 | 实现助教要求的 `ingest()` / `answer()` 接口 |

### 2.2 已完成的对照实验入口

当前已实现以下五组实验系统：

| 实验编号 | 系统 | 检索对象 | 更新策略 | 用途 |
|---|---|---|---|---|
| B0 | `NoMemoryAgent` | 无 | 无 | 无记忆 baseline |
| B1 | `FullContextAgent` | 完整历史对话 | 无 | 全上下文 baseline |
| B2 | `VanillaRAGAgent` | 原始对话 turn | 无 | 原始切片 RAG baseline |
| S1 | `AppendOnlyMemoryAgent` | 派生记忆单元 | 只追加 | 我方系统基线 |
| S2 | `ConflictAwareMemoryAgent` | 派生记忆单元 | 冲突感知更新 | 核心方案 |

其中，S2 在写入新记忆时会检索语义相近的活跃旧记忆。当相似度超过阈值后，调用 LLM
判断两条记忆之间属于 `NOOP`、`ADD`、`UPDATE` 或 `DELETE`。对于 `UPDATE`，旧记忆会
被标记为 `superseded`，新记忆保存 `supersedes` 字段和递增后的版本号。

### 3.3 已识别并解决的问题

开发过程中已解决以下问题：

| 问题 | 处理方式 |
|---|---|
| LoCoMo 官方数据格式与初始假设不一致 | 扩展数据加载器，兼容官方单文件格式和助教简化格式 |

| 助教工具包旧下载链接失效 | 根据正确 GitHub 路径下载并接入正式工具包 |
| 助教脚本在 Windows 下使用 GBK 写文件失败 | 对 vendored 副本补充 UTF-8 读写和目录创建 |

## 六、风险与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 云端 API 限流或网络波动 | 全集生成中断 | 使用 `--resume` 断点续跑；先执行小样本 smoke test |
| Embedding 或生成 API 配置不兼容 | 检索或生成失败 | 通过统一 OpenAI 兼容客户端封装；先验证 B0，再逐步验证 B2、S1、S2 |
| 冲突判断增加 LLM 调用成本 | S2 运行时间和费用上升 | 使用相似度阈值，仅对高相似候选调用 LLM |
| S2 分类均值提升不明显 | 难以展示更新机制收益 | 增加冲突事实专项 bad case 分析，结合分类指标解释效果 |
| Windows 环境依赖不稳定 | 无法使用 FAISS 或脚本编码错误 | 使用 NumPy 后备索引和 UTF-8 包装入口 |
