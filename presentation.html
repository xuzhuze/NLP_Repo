选题背景与目标

LLM 是无状态的——
对话一结束，它就把你忘了

  • 痛点：模型不记得跨会话的用户信息，长期交互中无法回忆、利用、更新历史事实。
  • 关键难题：旧信息会过时（"我搬到了上海"→后来"我去了北京"），单纯堆历史既塞不下也会自相矛盾。
  • 目标：从多会话对话提取记忆单元→检索辅助问答→在冲突时更新/替换旧记忆。
  • 课程关联：信息抽取、稠密检索(RAG)、Prompt 工程、LLM-as-Judge 评测的综合实践。

核心创新点

冲突感知更新：新事实到来时，让旧记忆"退役"

  • 写入新记忆前，先检索语义相近的活跃旧记忆。
  • 相似度超阈值后，调 LLM 判定四选一：
    NOOP / ADD / UPDATE / DELETE。
  • 判 UPDATE 时：旧记忆标记 superseded，新记忆记 supersedes + 版本号 +1。
  • 仅对高相似候选调 LLM，控制成本。

S2 − S1 = 冲突感知更新的净收益

写入耗时 S2/S1
  • 逐题对比：S2 相对 S1 4 题变好、5 题变差，但靠部分得分加权，总分仍净增 +1.1pp。小样本下效应温和。
  • 成本-效果权衡：冲突判断让 S2 写入耗时是 S1 的 ~4.2 倍（1449s vs 343s），收益与开销需一并评估。
  • 观察点：抽样集无单独"信息更新"类别，须结合分类指标 + 含冲突事实的样例分析，不能只看类别均值。

Bad case 三类定位

① 写入丢失

"Melanie 何时去露营?"（答案 June 2023）→ S2 答"记忆未提及计划时间"：时间事实没被 Writer 提取。

② 检索未命中

"Melanie 有哪些爱好?"（陶艺/露营/绘画/游泳）→ S2 召回到跑步、小提琴等无关记忆。

③ 生成未利用

"Caroline 信教吗?"（答案：有点）→ S2 过度否定答"完全不信"，没用好检索到的记忆。

已解决的难点 & 下一步

实验中的挑战与应对

  • LoCoMo 官方/助教两套数据格式 → 加载器双兼容。
  • Windows 装不了 faiss-cpu → NumPy 后备索引。
  • 助教脚本 GBK 写 emoji 崩溃、数字答案评分崩溃 → UTF-8 + 类型兼容修复。
  • Embedding API 单次上限 → 自动分批。

局限与未来改进

  • 局限：S2 冲突判断引入额外 LLM 调用，写入耗时显著上升（成本-效果权衡）。
  • 局限：抽样规模有限，结论需更大样本验证。
  • S3 加分方向：三因子检索（relevance × recency × importance）。
  • 遗忘曲线、周期性 reflection 生成高阶记忆。
